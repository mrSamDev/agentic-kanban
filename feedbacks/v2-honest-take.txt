 ### Root Issue

 Kanban has the right data model (SQLite + leases + history) but wrong interface surface. CLI is a debugging tool pretending to be an agent protocol. The real coordination should happen through the DB reads/writes
 + a lightweight batch API.

 Your prior review said it best: "solo-agent serial queue masquerading as multi-agent coordination engine." Still true at v0.1.11.

 ### What I'd Ship Next

 1. kanban task batch claim + kanban task batch complete — single SQL write, not 33 round-trips
 2. claim-next --respect-deps — skip tasks whose depends_on isn't DONE
 3. kanban status --json — so agents read progress programmatically, not squint at ASCII

 Your v2 session proved the subagent pattern can do real work. The board management just needs to catch up to what the subagents can actually deliver.
