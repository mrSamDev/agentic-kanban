Kanban: How Useful Was It?

 Honest assessment from using it end-to-end on this project:

 What worked well:
 - State machine enforced correctly — TODO→IN_PROGRESS (claim), IN_PROGRESS→IN_REVIEW/DONE, IN_REVIEW→DONE all enforced. Saw invalid state transition when trying to approve a TODO task directly.
 - Lease/heartbeat model solid — claim_next_task set lease_until, log_progress renewed it. Gave confidence others wouldn't steal tasks.
 - JSON everywhere — Every CLI command returned structured JSON. No table-only output paths. Made automated consumption trivial.
 - Skill files — Good context for agents. The MCP tool descriptions + skill docs made self-discovery possible.
 - Structured errors — {"error":"invalid state transition"}, {"error":"task not assigned to this agent"} — never crashed.

 What was friction:
 - Subagent isolation gap — Worker subagents spawned their own kanban task IDs (TASK-2 etc.) separate from my original dispatch (TASK-12 etc.). No way to merge or alias. Caused confusion tracking what was actually
   done.
 - No batch claim via MCP — Had to dispatch 14 tasks via sequential kanban task dispatch calls in bash. Would love a dispatch_task({tasks: [...], parallel: true}) MCP shape.
 - No MCP-level --help — Had to fall back to bash kanban --help to discover flags. MCP tool descriptions covered basic use but not flag details.
 - Can't transition unclaimed tasks — complete TASK-12 --agent manager rejected with "task not assigned". Had to approve TASK-2 (which the subagent claimed). Inconsistency between the two task sets was annoying.
 - No dependency enforcement visible — I don't know if the CLI enforces depends_on at all. Didn't test it.

 Verdict: 7/10 — The state machine, lease model, and JSON output are mature. The agent integration gap (subagent task isolation, batch operations, cross-board visibility) is the main pain point. For a solo agent
 working in one session, it's solid. For multi-agent delegation, the tool surface needs a bit more bridging.
